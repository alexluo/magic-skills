## 🎯 核心决策

### 1. 技术路线：MCP 协议为核心

**决策**：采用 **MCP (Model Context Protocol)** 作为核心集成协议

**架构图**：
```
┌─────────────────────────────────────────────────────────┐
│                   AI 工具层                              │
│  ┌────────┐ ┌──────────┐ ┌────────┐ ┌──────┐ ┌──────┐ │
│  │ Cursor │ │Claude D.T│ │VS Code │ │ Trae │ │ Open │ │
│  └────────┘ └──────────┘ └────────┘ └──────┘ └──────┘ │
└─────────────────────────────────────────────────────────┘
                          │
          ┌───────────────┼───────────────┐
          │               │               │
          ▼               ▼               ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│   MCP        │ │   LSP        │ │   REST       │
│   Server     │ │   Server     │ │   API        │
│   (主)       │ │   (辅)       │ │   (备用)     │
└──────┬───────┘ └──────┬───────┘ └──────┬───────┘
       │                │                │
       └────────────────┼────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│              Magic Skills 核心                           │
└─────────────────────────────────────────────────────────┘
```

**理由**：
- ✅ Cursor、Claude Desktop 原生支持 MCP
- ✅ 标准化协议，支持工具发现和参数自动补全
- ✅ 双向通信，AI 可以询问技能详情
- ✅ 原生支持上下文传递（读取编辑器内容）
- ✅ 适配器模式，便于未来支持新工具

## 🔧 核心功能设计

### 上下文读取架构

**场景示例**：代码审查

```
用户在 Cursor 中选中一段代码
↓
对 AI 说："帮我审查这段代码"
↓
AI 通过 MCP 调用 Magic Skills
  - 传递选中的代码（context.selection）
  - 传递文件路径（context.file）
  - 传递语言类型（context.language）
↓
code_reviewer 技能执行
  - 读取代码
  - 分析代码质量
  - 生成审查报告
↓
结果返回给 AI，AI 展示给用户
```

**MCP 上下文传递实现**：
```python
@mcp.call_tool()
async def execute_skill(name: str, args: dict, context: MCPContext):
    # 从上下文读取编辑器内容
    editor_context = {
        "selected_code": context.selection.text,
        "file_path": context.document.uri,
        "language": context.document.language_id,
        "full_file": await context.document.get_text()
    }
    
    # 执行技能
    result = await skill_manager.execute_skill(
        name,
        {**args, "context": editor_context}
    )
    
    return result
```

## ⚠️ 风险评估

### 风险 1：AI 工具集成复杂度超预期

**风险**：
- MCP 协议相对新颖，文档和社区支持有限
- VS Code 扩展开发需要 TypeScript 技能
- 不同 AI 工具的插件系统差异大

**缓解措施**：
- 分阶段实施：先完成 Cursor/Claude Desktop
- 简化 MVP：VS Code 扩展先实现基础功能
- 技术预研：提前调研 Trae/OpenCode

### 风险 2：开发周期延长

**风险**：
- AI 工具集成增加大量工作量
- VS Code 扩展开发可能遇到技术难题

**缓解措施**：
- 优先级排序：
  - P0（MVP 必需）：MCP Server、Cursor、Claude Desktop
  - P1（重要）：VS Code 扩展基础功能
  - P2（可延后）：Trae/OpenCode、高级功能
- 预留 1 周缓冲时间

### 风险 3：用户采用门槛

**风险**：
- 配置多个 AI 工具，用户可能觉得复杂
- MCP Server 配置涉及命令行

**缓解措施**：
- 一键配置：`magic-skill init --ai-tool=cursor`
- 详细文档：图文教程 + 视频教程
- 预配置文件：提供模板

## 💡 关键洞察

### 1. AI 工具集成是**战略性决策**

这不仅仅是技术集成，更是**用户体验的革命**：
- 从"打开 CLI → 输入命令"到"在编辑器中直接使用"
- 从"手动配置"到"AI 自动推荐和调用"
- 从"单一工具"到"AI 工具 + 技能生态系统"

### 2. MCP 协议是**关键赌注**

选择 MCP 作为核心协议的理由：
- ✅ 标准化：避免为每个工具重复实现
- ✅ 未来proof：MCP 生态正在快速增长
- ✅ 上下文原生支持：这是 REST API 无法比拟的

但也需要**风险对冲**：
- 保留 REST API 作为备选
- 使用适配器模式，便于切换协议

### 3. 上下文管理是**核心竞争力**

读取编辑器上下文不仅仅是"传递代码"：
- **智能压缩**：减少 Token 消耗，降低成本
- **AST 解析**：理解代码结构，提供更精准的服务
- **多文件引用**：跨文件分析，理解项目全局
- **缓存优化**：提升响应速度

这些功能将决定 Magic Skills 与简单 API 调用的**差异化价值**。

### 4. VS Code 扩展是**最大挑战**

- 需要 TypeScript 技能（如果团队不熟悉）
- VS Code API 复杂，学习曲线陡峭
- 但也是**最大的用户群**（VS Code 有 1400 万开发者）

**建议**：
- 如果时间紧张，可以先实现基础功能（技能面板、右键菜单）
- 高级功能（Code Lens、Code Action）可以放到 v1.1
