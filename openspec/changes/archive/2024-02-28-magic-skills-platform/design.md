## Context

### 背景
当前 AI 应用开发面临三大挑战：
1. **LLM 集成碎片化**：每个 LLM 提供商有不同的 API、SDK 和认证方式，开发者需要重复编写适配代码
2. **技能复用困难**：缺乏统一的技能管理和分享机制，每个项目都需要从头实现
3. **缺乏智能优化**：Prompt 和参数调优依赖人工经验，缺乏自动化的性能提升机制

### 当前状态
- 项目位置：`/Users/youlianyao/xdot/my_agents/magic_skills`
- OpenSpec 配置：`spec-driven` 模式
- 技术栈：Python 3.10+
- 部署方式：本地运行（CLI + REST API）

### 约束条件
- **时间约束**：3-4 周完成 MVP
- **技术约束**：Python 生态，本地部署
- **资源约束**：单人开发，社区贡献为辅
- **性能约束**：技能执行延迟 < 3 秒（不含 LLM 响应）

### 利益相关者
- **开发者**：使用 API/SDK 集成技能到应用
- **最终用户**：通过 CLI 或 UI 使用技能
- **贡献者**：开发新技能或优化现有功能
- **学习者**：学习项目代码和架构设计

## Goals / Non-Goals

**Goals:**
- ✅ 实现统一的 LLM 抽象层，支持 10+ 个主流模型（OpenAI、Claude、Gemini、通义千问等）
- ✅ 构建基于文件系统的技能管理系统，支持技能的加载、执行和版本控制
- ✅ 开发自我优化引擎，实现 Prompt 优化和参数自动调优
- ✅ 提供 CLI 工具和 REST API 双接口，满足开发者和最终用户需求
- ✅ 建立完整的开源基础设施（文档、测试、CI/CD、示例）
- ✅ 实现反馈收集系统，支持用户反馈和日志分析

**Non-Goals:**
- ❌ 云端 SaaS 服务部署（聚焦本地运行）
- ❌ 图形化 UI 界面（v2 阶段考虑）
- ❌ 分布式技能执行（v3 阶段考虑）
- ❌ 技能市场/注册表（v2 阶段考虑）
- ❌ 多语言支持（仅 Python，其他语言绑定可作为社区贡献）
- ❌ 实时流式响应（MVP 阶段仅支持同步调用，流式作为 v1.1）

## Decisions

### 1. 技术栈选择

**决策**：使用 Python 3.10+ 作为主要开发语言

**理由**：
- AI/ML 生态丰富（LangChain、LlamaIndex、Transformers）
- LLM SDK 支持完善（所有主流模型都有 Python SDK）
- 快速原型开发能力
- 易于学习和贡献

**替代方案**：
- Node.js/TypeScript：生态好但 AI 库相对较少
- Go/Rust：性能好但生态不成熟

---

### 2. 架构模式

**决策**：分层架构（Layered Architecture）

```
┌─────────────────────────────────────────┐
│         接口层（CLI + REST API）         │
├─────────────────────────────────────────┤
│         应用服务层（Manager）            │
├─────────────────────────────────────────┤
│         核心逻辑层（Core）               │
├─────────────────────────────────────────┤
│         数据访问层（Providers）          │
└─────────────────────────────────────────┘
```

**理由**：
- 职责清晰，易于维护和测试
- 符合 Python 社区惯例
- 便于新人理解和贡献

**替代方案**：
- 六边形架构：过度复杂，不适合 MVP
- 事件驱动：增加复杂度，暂不需要

---

### 3. LLM 集成策略

**决策**：Level 2 统一接口抽象

```python
class BaseLLMProvider(ABC):
    @abstractmethod
    async def generate(self, prompt: str, options: dict) -> str:
        pass
    
    @abstractmethod
    async def stream(self, prompt: str, options: dict) -> AsyncIterator[str]:
        pass
```

**理由**：
- 平衡抽象度和实现复杂度
- 上层应用无需关心具体模型
- 便于未来升级到 Level 3（智能路由）

**替代方案**：
- Level 1（无抽象）：代码重复，难以维护
- Level 3（智能路由）：MVP 阶段过度设计

---

### 4. 技能存储方案

**决策**：文件系统存储（YAML + Python）

每个技能是一个目录：
```
skills/hello_world/
├── skill.yaml      # 技能配置（名称、描述、参数）
├── prompt.txt      # Prompt 模板
└── handler.py      # 处理逻辑
```

**理由**：
- 简单直观，易于理解和贡献
- Git 版本控制友好
- 技能包易于分享和分发
- 符合开源项目惯例

**替代方案**：
- 数据库存储：增加部署复杂度
- 混合模式：MVP 阶段过度设计

---

### 5. 自我优化实现策略

**决策**：分阶段实现

**阶段 1（MVP，v1.0）**：
- 基础反馈收集（SQLite 存储）
- Prompt 版本管理（文件系统）
- 简单日志分析（JSONL 文件）

**阶段 2（v1.1）**：
- A/B 测试框架
- 参数自动调优（贝叶斯优化）
- 模型选择推荐

**阶段 3（v2.0）**：
- 强化学习优化
- 智能模型路由
- 自动技能组合

**理由**：
- 降低 MVP 复杂度
- 快速验证核心价值
- 预留扩展空间

---

### 6. API 设计

**决策**：RESTful API + CLI 双接口

**REST API**（FastAPI）：
```python
POST /api/skills/execute
GET  /api/skills/list
POST /api/models/list
POST /api/feedback/submit
```

**CLI**（Typer）：
```bash
magic-skill exec hello_world --name World
magic-skill list
magic-skill optimize hello_world
```

**理由**：
- 满足不同用户群体需求
- REST API 便于集成
- CLI 便于调试和直接使用

---

### 7. 数据模型

**决策**：轻量级数据模型

**反馈数据**（SQLite）：
```sql
CREATE TABLE feedback (
    id INTEGER PRIMARY KEY,
    skill_name TEXT,
    execution_id TEXT,
    rating INTEGER,  -- 1 or -1
    comment TEXT,
    created_at TIMESTAMP
);
```

**执行日志**（JSONL 文件）：
```json
{"timestamp": "...", "skill": "hello_world", "model": "gpt-4", "duration_ms": 1234, "tokens": 150}
```

**理由**：
- SQLite 零配置，适合本地运行
- JSONL 便于日志分析
- 避免重型数据库依赖

**理由**：

### 11. AI 工具集成策略

**决策**：以 MCP 协议为核心，统一协议 + 适配器模式

```
┌─────────────────────────────────────────────────────────┐
│                   AI 工具层                              │
│  ┌────────┐ ┌──────────┐ ┌────────┐ ┌──────┐ ┌──────┐ │
│  │ Cursor │ │Claude D.T│ │VS Code │ │ Trae │ │ Open │ │
│  └────────┘ └──────────┘ └────────┘ └──────┘ └──────┘ │
└─────────────────────────────────────────────────────────┘
                          │
          ┌───────────────┼───────────────┐
          │               │               │
          ▼               ▼               ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│   MCP        │ │   LSP        │ │   REST       │
│   Server     │ │   Server     │ │   API        │
│   (主)       │ │   (辅)       │ │   (备用)     │
└──────┬───────┘ └──────┬───────┘ └──────┬───────┘
       │                │                │
       └────────────────┼────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│              Magic Skills 核心                           │
└─────────────────────────────────────────────────────────┘
```

**理由**：
- **MCP 协议**：Cursor、Claude Desktop 原生支持，标准化协议
- **统一架构**：避免为每个工具重复实现
- **适配器模式**：便于未来支持新工具
- **上下文读取**：MCP 原生支持传递编辑器内容

**替代方案**：
- 方案 A：为每个工具开发专用插件 → 维护成本过高
- 方案 B：仅支持 REST API → 体验割裂，无法读取上下文
- 方案 C：等待所有工具支持 MCP → 失去市场先机

**实施细节**：
- **Cursor/Claude Desktop**：MCP 原生支持，配置文件即可
- **VS Code**：开发专用扩展（TypeScript），通过 LSP 或 MCP 通信
- **Trae/OpenCode**：调研后决定（MCP 优先，专用插件备选）

### 12. 上下文管理设计

**决策**：多层上下文提取和压缩机制

**上下文层次**：
```
Level 1: 选中文本（最常用）
Level 2: 完整文件
Level 3: 项目结构
Level 4: 多文件引用
Level 5: AST 解析结果
```

**上下文压缩策略**：
- 移除空白行和注释（可配置）
- 智能摘要（保留关键代码）
- Token 计数优化
- 缓存机制（避免重复读取）

**理由**：
- 减少 Token 消耗，降低成本
- 提升响应速度
- 保留关键信息，不影响技能执行

---

### 13. 技能与 AI 工具的交互模式

**决策**：双向智能工作流

**模式 1：AI 驱动**
```
用户 → AI 工具 → MCP → Magic Skills → 执行结果 → AI 工具 → 用户
```

**模式 2：用户驱动**
```
用户 → VS Code 扩展 → Magic Skills → 执行结果 → 输出面板
```

**模式 3：链式调用**
```
技能 A → 结果分析 → 条件判断 → 技能 B → 最终结果
```

**理由**：
- 满足不同使用习惯
- 提升用户体验
- 支持复杂场景


## Risks / Trade-offs

### 风险 1：LLM API 成本和可用性

**风险**：
- 多个 LLM API 需要多个 API 密钥，增加用户门槛
- API 调用成本可能较高
- 某些 LLM 服务可能不稳定或受限（特别是国内 LLM）

**缓解措施**：
- 支持本地模型（Ollama）作为备选
- 提供 Mock 模式用于开发测试
- 文档明确标注每个 LLM 的成本和稳定性

---

### 风险 2：自我优化效果不达预期

**风险**：
- Prompt 优化算法复杂，效果难以保证
- 需要大量数据才能训练有效的优化模型
- 可能陷入局部最优

**缓解措施**：
- MVP 阶段聚焦基础功能（反馈收集、简单统计）
- 采用成熟的优化算法（贝叶斯优化）
- 明确标注"实验性功能"，管理用户预期

---

### 风险 3：AI 工具集成复杂度超预期

**风险**：
- MCP 协议相对新颖，文档和社区支持有限
- VS Code 扩展开发需要 TypeScript 技能，增加学习成本
- 不同 AI 工具的插件系统差异大，适配难度高
- 上下文读取可能涉及编辑器 API 限制

**缓解措施**：
- **分阶段实施**：先完成 Cursor/Claude Desktop（MCP 原生支持）
- **简化 MVP**：VS Code 扩展先实现基础功能，高级功能后续迭代
- **技术预研**：提前调研 Trae/OpenCode 的扩展机制，评估可行性
- **社区协作**：如有可能，邀请有编辑器插件开发经验的贡献者

---

### 风险 4：MCP 协议的长期可持续性

**风险**：
- MCP 由 Anthropic 主导，未来发展方向不确定
- 如果其他 AI 工具不支持 MCP，需要额外适配
- 协议变更可能导致兼容性问题

**缓解措施**：
- **适配器模式**：抽象协议层，便于切换到其他协议
- **REST API 备选**：即使不支持 MCP，也能通过 REST API 集成
- **关注生态**：密切关注 MCP 生态发展，及时调整策略

---

### 风险 5：开发周期延长

**风险**：
- AI 工具集成增加大量工作量（原 3-4 周 → 现在 4 周 +）
- VS Code 扩展开发可能遇到技术难题
- 多个工具测试和调试耗时

**缓解措施**：
- **优先级排序**：
  - P0（MVP 必需）：MCP Server、Cursor、Claude Desktop 集成
  - P1（重要）：VS Code 扩展基础功能
  - P2（可延后）：Trae/OpenCode 集成、高级功能
- **时间缓冲**：预留 1 周缓冲时间应对意外
- **功能裁剪**：如果时间紧张，VS Code 扩展先实现核心功能（技能面板、右键菜单）

---

### 风险 6：用户采用门槛

**风险**：
- 需要配置多个 AI 工具，用户可能觉得复杂
- MCP Server 配置涉及命令行，非技术用户可能困难
- 不同 AI 工具的配置方式不同，容易混淆

**缓解措施**：
- **一键配置**：提供 `magic-skill init --ai-tool=cursor` 等命令
- **图形化配置工具**：考虑开发简单的配置 GUI（可选）
- **详细文档**：每个工具的图文教程 + 视频教程
- **预配置文件**：提供模板，用户只需修改少量参数


---

### 风险 3：性能瓶颈

**风险**：
- 文件系统加载技能可能较慢
- 多个 LLM 并发调用可能超时
- 日志分析可能占用大量内存

**缓解措施**：
- 实现技能缓存机制
- 使用异步 IO（asyncio）
- 日志轮转和压缩
- 性能基准测试和持续优化

---

### 风险 4：维护复杂度

**风险**：
- 10+ 个 LLM SDK 需要持续维护
- SDK 版本更新可能导致兼容性问题
- 国内 LLM API 可能频繁变更

**缓解措施**：
- 每个 LLM 提供商独立的模块和测试
- 依赖版本锁定（requirements.txt）
- CI/CD 自动测试所有 LLM 集成
- 社区维护机制（每个 LLM 一个维护者）

---

### 风险 5：时间延期

**风险**：
- 3-4 周时间框架紧张
- 自我优化功能可能超出预期复杂度
- 文档和测试耗时可能低估

**缓解措施**：
- 优先级排序（核心功能 > 增强功能）
- MVP 范围严格控制
- 自我优化分阶段实现
- 文档和测试与开发同步进行

---

### 风险 6：社区冷启动

**风险**：
- 开源项目可能缺乏关注和贡献
- 技能生态发展缓慢
- 缺乏持续维护者

**缓解措施**：
- 高质量文档和示例
- 积极的社区运营（Twitter、Reddit、知乎等）
- 降低贡献门槛（Good First Issue 标签）
- 寻找共同维护者
